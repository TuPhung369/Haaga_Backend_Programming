Database Design

Detailed documentation of the project's database schema, relationships, and design patterns.

Overview

The application uses a relational database management system (PostgreSQL) to store and manage data. The database schema is designed following best practices for data modeling, including appropriate use of primary keys, foreign keys, indexes, and constraints.

Design Principles

  • Normalization - The database is normalized to reduce redundancy and improve data integrity
  • Referential Integrity - Foreign key constraints ensure data consistency
  • Indexing Strategy - Strategic indexes for optimized query performance
  • Auditing - Creation and modification timestamps on all entities
  • UUID Primary Keys - Use of UUIDs instead of sequential IDs for security and distribution

Data Access Layer

JPA Entity Example

Example of a JPA entity class with annotations:

@Entity
@Table(name = "users")
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class User {
    
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;
    
    @Column(unique = true, nullable = false)
    private String email;
    
    @Column(nullable = false)
    private String password;
    
    @Column(nullable = true)
    private String username;
    
    @Column(nullable = false)
    @Enumerated(EnumType.STRING)
    private UserStatus status;
    
    @ManyToMany(fetch = FetchType.EAGER)
    @JoinTable(
        name = "user_roles",
        joinColumns = @JoinColumn(name = "user_id"),
        inverseJoinColumns = @JoinColumn(name = "role_id")
    )
    private Set roles = new HashSet<>();
    
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL, orphanRemoval = true)
    private List events = new ArrayList<>();
    
    @CreationTimestamp
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;
    
    @UpdateTimestamp
    @Column(name = "updated_at")
    private LocalDateTime updatedAt;
}

Repository Interface Example

Example of a Spring Data JPA repository interface:

@Repository
public interface UserRepository extends JpaRepository {
    
    Optional findByEmail(String email);
    
    boolean existsByEmail(String email);
    
    @Query("SELECT u FROM User u WHERE u.status = :status")
    List findByStatus(@Param("status") UserStatus status);
    
    @Query(value = "SELECT u.* FROM users u " +
           "JOIN user_roles ur ON u.id = ur.user_id " +
           "JOIN roles r ON ur.role_id = r.id " +
           "WHERE r.name = :roleName", 
           nativeQuery = true)
    List findByRoleName(@Param("roleName") String roleName);
}

Database Migration and Versioning

Flyway Migrations

The project uses Flyway for database migrations to manage schema changes over time. Migration scripts follow the naming convention V[version]__[description].sql.

Example Migration Script

-- V1__initial_schema.sql

CREATE TABLE users (
    id UUID PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    password VARCHAR(255) NOT NULL,
    username VARCHAR(255),
    status VARCHAR(50) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP
);

CREATE TABLE roles (
    id UUID PRIMARY KEY,
    name VARCHAR(100) NOT NULL UNIQUE,
    description VARCHAR(255)
);

CREATE TABLE user_roles (
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    role_id UUID NOT NULL REFERENCES roles(id),
    PRIMARY KEY (user_id, role_id)
);

Performance Considerations

Indexing Strategy

The database uses appropriate indexes to optimize query performance:

  • Primary Key Indexes - Automatically created on all primary key columns
  • Foreign Key Indexes - Created on foreign key columns to optimize joins
  • Search Column Indexes - Added to columns frequently used in WHERE clauses
  • Composite Indexes - Used for multi-column searches and sorting

Example Index Creation

-- Create index on email for fast user lookups
CREATE INDEX idx_users_email ON users(email);

-- Create index on role name for role lookups
CREATE INDEX idx_roles_name ON roles(name);

-- Create composite index for message filtering
CREATE INDEX idx_chat_messages_type_sent_at ON chat_messages(type, sent_at DESC);

Caching Strategy

The application implements multi-level caching to reduce database load:

  • First-Level Cache - Hibernate's session-level cache for entity caching
  • Second-Level Cache - Caffeine cache for frequently accessed entities
  • Query Cache - Caching of query results for commonly executed queries

Example Cache Configuration

@Configuration
@EnableCaching
public class CacheConfig {
    
    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .expireAfterWrite(30, TimeUnit.MINUTES)
            .maximumSize(500));
        return cacheManager;
    }
}
